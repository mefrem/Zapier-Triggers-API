## Context and Background

This story delivers the developer experience layer for the Zapier Triggers API. While Stories 1.2-1.7 focused on API functionality and Story 1.8 on operational observability, this story ensures developers can discover, understand, and implement integrations quickly.

Clear, complete documentation reduces support burden, enables self-service integration, and improves adoption. Sample clients in popular languages (Python, Node.js) demonstrate best practices and accelerate integration.

Key documentation deliverables:
1. **OpenAPI Specification:** Machine-readable API contract (Swagger/OpenAPI 3.0)
2. **Getting Started Guide:** 5-minute setup walkthrough
3. **API Reference:** Complete endpoint documentation with examples
4. **Sample Clients:** Production-ready Python and Node.js implementations
5. **Error Handling Guide:** All error codes and recovery strategies
6. **Rate Limiting Guide:** Best practices for handling limits

**Dependencies:**
- Story 1.2: Event Ingestion Endpoint - POST /events documentation
- Story 1.3: Authentication and Authorization - API key setup
- Story 1.4: Event Storage with DynamoDB - Data model documentation
- Story 1.5: Event Inbox Endpoint - GET /inbox documentation
- Story 1.6: Event Acknowledgment and Deletion - DELETE/ACK documentation
- Story 1.7: Retry and Status Tracking - Retry behavior documentation

- Must be valid and testable with tools like Swagger UI or ReDoc
- Example schema structure:
```yaml
openapi: 3.0.0
info:
  title: Zapier Triggers API
  version: 1.0.0
  description: Event delivery API for Zapier
  contact:
    name: Zapier API Support
    email: api-support@zapier.com

  - url: https://api.zapier.com/v1
    description: Production
  - url: https://staging-api.zapier.com/v1
    description: Staging

paths:
  /events:
    post:
      summary: Ingest an event
      description: |
        Create a new event for a Zapier workflow.
        Events are stored and made available via GET /inbox.
      tags:
        - Events
      security:
        - ApiKeyAuth: []
      requestBody:
        required: true
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/EventInput'
      responses:
        '201':
          description: Event successfully ingested
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/EventResponse'
        '401':
          $ref: '#/components/responses/Unauthorized'
        '429':
          $ref: '#/components/responses/RateLimited'

  /inbox:
    get:
      summary: Get undelivered events
      description: Retrieve events not yet acknowledged by workflows.
      tags:
        - Events
      security:
        - ApiKeyAuth: []
      parameters:
        - name: limit
          in: query
          schema:
            type: integer
            minimum: 1
            maximum: 100
            default: 10
        - name: cursor
          in: query
          schema:
            type: string
      responses:
        '200':
          description: Events successfully retrieved
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/InboxResponse'
```

### 6. Error Handling Guide with All Error Codes
**Given** a developer encounters an error
**When** they reference the error handling guide
**Then** they find:
- Error code explanation
- Likely causes
- Resolution steps
- Example response

**Implementation Notes:**
- Location: `docs/error-handling.md`
- Format: Searchable reference of all error codes
- Error categories:
  1. **4xx Client Errors:** Invalid request, auth failure, rate limit
  2. **5xx Server Errors:** Transient failures, temporary outages
- Error reference table:
```markdown
## Error Reference

| Code | HTTP | Message | Cause | Resolution |
|------|------|---------|-------|-----------|
| INVALID_PAYLOAD | 400 | Payload validation failed | Missing required field or wrong type | Check event_type and payload structure |
| PAYLOAD_TOO_LARGE | 400 | Payload size exceeds 1MB | Event payload >1MB | Reduce payload size or split event |
| INVALID_EVENT_TYPE | 400 | Event type is invalid | event_type not in format "domain.action" | Use format like "user.created", "order.paid" |
| UNAUTHORIZED | 401 | Invalid or missing API key | Missing X-API-Key header or invalid key | Verify API key is set correctly |
| RATE_LIMITED | 429 | Rate limit exceeded (1000/min) | Too many requests in 1 minute | Implement exponential backoff, spread requests |
| NOT_FOUND | 404 | Resource not found | Event ID doesn't exist or belongs to different user | Verify event ID is correct |
| SERVER_ERROR | 500 | Unexpected server error | Temporary service issue | Retry with exponential backoff |
```
- Retry strategy guide:
```markdown
## Retry Strategy

### When to Retry
- 429 (Rate Limited) - Always retry
- 5xx (Server Error) - Always retry with backoff
- 4xx (Client Error) - Usually don't retry (fix error first)
- 401 (Unauthorized) - Don't retry (fix API key)

### Recommended Backoff
- Attempt 1: Immediate
- Attempt 2: 100ms delay
- Attempt 3: 500ms delay
- Attempt 4: 2s delay
- Attempt 5: 5s delay
- Max 5 attempts total

### Example Implementation (Python)
\`\`\`python
import time

def ingest_with_retry(client, event, max_attempts=5):
    for attempt in range(max_attempts):
        try:
            return client.ingest_event(event)
        except RateLimitError:
            delay = 2 ** attempt * 0.1  # Exponential backoff
            print(f"Rate limited. Waiting {delay}s before retry...")
            time.sleep(delay)
        except ServerError:
            if attempt < max_attempts - 1:
                delay = 2 ** attempt * 0.1
                print(f"Server error. Retrying in {delay}s...")
                time.sleep(delay)
            else:
                raise
\`\`\`
```

### 7. Rate Limiting Documentation with Best Practices
**Given** a developer wants to handle rate limiting
**When** they reference the rate limiting guide
**Then** they understand:
- Rate limit policy (1000 req/min)
- Headers returned (X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset)
- How to detect approaching limit
- Best practices for scaling

**Implementation Notes:**
- Location: `docs/rate-limiting.md`
- Content includes:
```markdown
## Rate Limiting

### Policy
- Limit: 1000 requests per minute per API key
- Window: Rolling 1-minute window
- Applies to: All endpoints (POST /events, GET /inbox, etc.)
- Burst: Up to 50 requests allowed in single second (brief bursts)

### Rate Limit Headers
Every response includes:
- X-RateLimit-Limit: 1000 (limit per minute)
- X-RateLimit-Remaining: 995 (requests remaining this minute)
- X-RateLimit-Reset: 1699691860 (Unix timestamp when limit resets)

### Detecting Approaching Limit
Check X-RateLimit-Remaining header:
- >500: Safe to proceed
- 100-500: Consider slowing down
- <100: Implement throttling
- 0: Stop and wait for window reset

### Handling 429 Response
\`\`\`json
{
  "error": {
    "code": "RATE_LIMITED",
    "message": "Rate limit exceeded (1000/min)",
    "timestamp": "2025-11-11T10:00:00Z",
    "request_id": "req-123"
  },
  "headers": {
    "X-RateLimit-Reset": 1699691860,
    "Retry-After": 55
  }
}
\`\`\`

### Best Practices for Scaling
1. **Batch Requests:** Combine multiple events into single request when possible
2. **Throttle:** Spread requests evenly over time (e.g., 100 req/sec not 1000 in 1 sec)
3. **Monitor:** Track X-RateLimit-Remaining header
4. **Request Limit Increase:** Contact support@zapier.com if you need higher limits

### Example Throttling (Python)
\`\`\`python
import time
from rate_limiter import RateLimiter

limiter = RateLimiter(rate_per_minute=900)  # Leave 100-req buffer

for event in events:
    limiter.wait()  # Sleeps if necessary
    client.ingest_event(event)
\`\`\`
```

- Contact process: Who to contact for limit increases

**Risk: Documentation Out-of-Sync with Code**

- Probability: Medium (if not versioned with code)
- Mitigation: Version docs with each code release, include in CI tests

**Risk: Sample Clients Unused/Unmaintained**
- Impact: Outdated code, confused developers
- Probability: Low (if clearly marked as samples)
- Mitigation: Include deprecation warnings, provide links to official SDKs
- Testing: Regular reviews and updates alongside API changes

**Risk: Poor Developer Experience (UX)**
- Impact: Long integration time, support burden
- Probability: Medium (subjective)
- Mitigation: Get external feedback on docs, iterate based on feedback
- Testing: Usability testing with target developers
